# Phase 03.2: Backend Integration & Rate Limiting - Research

## Context Summary
The primary objective of this phase is closing integration gaps. Specifically, the v1.0 Milestone Audit noted two failures in Phase 2:
- `API-04`: Google OAuth was not configured correctly.
- `SEC-01`: Rate limiting was set to generic defaults (30 per hour), rather than the explicit 5 attempts per 15 min constraint.

## Domain Analysis
1. **Google OAuth Config**: To resolve this, `[auth.external.google]` needs to be activated inside `supabase/config.toml`, supplying placeholder secrets / setup keys (`client_id`, `secret`, `redirect_uri`).
2. **Rate Limiting**: `sign_in_sign_ups` config under the `[auth]` section of `supabase/config.toml` natively configures the max amount of signs in a specific frequency period. 15 minutes is 900 seconds. We map this back to 5 attempts.

## Technical Strategy
We will modify the `supabase/config.toml` structure. Standard Supabase configs:
- OAuth: `[auth.external.google]` with keys.
- Login Limits: `[auth]` under `[auth.email]` or globally under `[auth]` like `[auth.sms]`. But rate limiting limits usually exist globally on the API or specifically for `auth`. There are built-in GoTrue parameters natively configurable by Supabase CLI like `[auth.hook]` or via config values inside `[auth]`. 

## Validation Architecture
- **Criteria**: Checking `supabase/config.toml` explicitly via git to ensure the correct flags and integer representations are placed inside it. Checking the GoTrue server configuration syntax locally.
