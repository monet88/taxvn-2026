# Plan 03.2-01: Configure Backend OAuth & Limits

<objective>
Update `supabase/config.toml` to explicitly configure Google OAuth client parameters (resolving API-04) and enforce the auth rate limit function limiting signups/signins to 5 attempts per 15 minutes (resolving SEC-01).
</objective>

<tasks>

<task>
  <description>Configure Google OAuth Settings.</description>
  <read_first>
    - supabase/config.toml
  </read_first>
  <action>
    Add the `[auth.external.google]` block to `supabase/config.toml`.
    Ensure `enabled = true`.
    Configure `client_id` and `secret` using `env` variables, falling back to a dummy string locally if not present. Use standard Supabase config syntax:
    ```toml
    [auth.external.google]
    enabled = true
    client_id = "env(GOOGLE_CLIENT_ID)"
    secret = "env(GOOGLE_CLIENT_SECRET)"
    redirect_uri = "env(SUPABASE_AUTH_EXTERNAL_GOOGLE_REDIRECT_URI)"
    ```
  </action>
  <acceptance_criteria>
    - `supabase/config.toml` contains `[auth.external.google]` followed by `enabled = true`.
    - `supabase/config.toml` contains `client_id = "env(GOOGLE_CLIENT_ID)"` and the associated secret parameter.
  </acceptance_criteria>
</task>

<task>
  <description>Enforce 5 Attempts / 15 Minute Rate Limiting.</description>
  <read_first>
    - supabase/config.toml
  </read_first>
  <action>
    Modify or add rate limiting constraints in the `[auth]` section. Supabase's local config rate limit keys are usually within GoTrue block definitions or within `[auth]`. GoTrue enforces `limit_email_otp` / `limit_sms_otp` / global requests depending on version. Wait, this app primarily uses password auth. The Supabase parameter we need is usually `[auth]` rate limiting:
    ```toml
    [auth.rate_limit]
    email_sent = 5
    ```
    However, the goal requires limiting signins/signups to 5 attempts per 15 mins. Supabase CLI v1 handles API rate limits natively via GoTrue: `MFA_MAX_ENROLLED_FACTORS`, `RATE_LIMIT_TOKEN_REFRESH`, `RATE_LIMIT_VERIFY`, configuration limits inside config.toml are likely `[auth]` block parameters like `[auth.rate_limit.token_refresh]` or generic API gateway limits.
    Actually, per Supabase docs, password sign-in rate limitations are handled across Cloud parameters but locally we modify `[auth]` to:
    ```toml
    [auth]
    # For email sign in/up specifically
    sign_in_sign_ups = { limit = 5, interval = 900 }
    # Or simply adapt whatever the equivalent `supabase start` syntax dictates.
    ```
    Assuming standard syntax is used (which in many iterations is simply setting the env var `RATE_LIMIT_TOKEN_REFRESH=5`, `RATE_LIMIT_EMAIL_SENT=5`), adjust `supabase/config.toml` to enforce a 5 times / 15 minutes (900 seconds) cap. Typically:
    ```toml
    [auth.email]
    # some properties
    
    [auth.rate_limit]
    # (if applicable for GoTrue)
    ```
    Wait, the `REQUIREMENTS.md` mentions: `Login rate limiting (max 5 attempts/15 min per IP, exponential backoff)`.
    So add the necessary lines equivalent to the 5 / 900 configuration limit in `supabase/config.toml` that Supabase GoTrue currently supports, or define it explicitly via API Gateway directives.
    For this task, specifically target the rate limit params inside `[auth]`. Ensure `rate_limit` (or related `limit`) equates to 5 attempts.
  </action>
  <acceptance_criteria>
    - `supabase/config.toml` contains a rate limit declaration equating to 5 attempts.
    - Rate Limit interval relates to 15 mins (e.g. 900 or 15).
  </acceptance_criteria>
</task>

</tasks>
